Operations & Lifecycle

How fixes work, retries, PR monitoring, AI questions, regression detection, cascading fixes, and deployment.

How a fix works

When Claudear picks up an issue, it goes through these stages:

  • Pending: picked up and queued
  • Success: the AI created a pull request
  • Merged: the PR was merged
  • Closed: the PR was closed without merging (triggers retry)
  • Failed: the attempt failed (triggers retry)
  • Cannot Fix: retry budget exhausted; issue is flagged for manual review
Pending → Success → Merged → Resolved
   |          \
   |           → Closed → retry
   v
 Failed → retry → Cannot Fix

Retries

If a fix fails or the PR is closed, Claudear automatically retries with exponential backoff. Configure max_retries, base_delay_ms, and max_delay_ms in the [retry] section of your config file.

You can see queued retries with claudear retries list and trigger them immediately with claudear retries process. The dashboard also shows pending and completed retries.

PR monitoring

Claudear polls for the status of every PR it opens. When a PR merges, it can automatically close the original issue — set auto_resolve_on_merge = true in your SCM configuration.

The polling interval is configured in your SCM section.

Answering AI questions

Sometimes the AI needs clarification before it can write a fix — unclear requirements, ambiguous code, or missing context. When this happens, Claudear sends you a question through your configured notification channels (Discord, Slack, or email). Just reply and the fix resumes.

Configure this behaviour in the [ask] section:

  • Set a timeout so Claudear doesn't wait forever
  • Enable best_effort_on_timeout to have Claudear make its best guess if no reply arrives
  • Discord and Slack are the easiest channels for replies: email works for notification only

Claudear remembers previous answers, so it won't ask the same question twice.

Regression monitoring

After a fix is merged, Claudear keeps watching for signs the problem came back. If the same error reappears in Sentry above a configured threshold, it's flagged as a regression in the dashboard.

Configure this in the [regression] section:

  • Set the monitoring window (how long to watch after a merge)
  • Set the check interval (how often to poll)
  • Set Sentry event thresholds and similarity matching

Use the diagnostic commands claudear diag release-graph, claudear diag release-check, and claudear diag release-path to verify that fixes are propagating through your release chain.

Cascading fixes across repos

If you have a library and an application that depends on it, enabling cascade means a fix in the library can automatically trigger a follow-up update in the application.

Configure relationships in the [cascade] section and use claudear repos link to declare dependencies. You can also auto-discover dependencies with claudear repos discover.

Set max_depth to control how deep the chain goes. Run claudear repos cascade <repo> to preview what would cascade from a given repo.

Learning from results

Over time, Claudear learns from successful fixes, PR review feedback, and Q&A answers. This knowledge is stored per-repo and used to improve future prompts — so fixes get better the longer Claudear runs.

Configure in the [learning] section. Key capabilities:

  • Analyses diffs and review comments from merged PRs
  • Promotes useful Q&A answers into reusable context
  • Scores strategies based on outcomes
  • Detects clusters of similar issues

Enable auto_agent_md = true to automatically write an AGENT.md file from accumulated knowledge, giving the AI a head start on every new fix.

Issue prioritisation

When many issues arrive at once, Claudear uses a scoring system to decide what to work on first. Signals include:

  • Error severity and frequency
  • Blast radius (authentication failures rank higher than cosmetic bugs)
  • Whether the issue clusters with other related issues

Configure weights and suppression rules in the [prioritisation] section. If you want to skip known-noisy issues (flaky tests, bot noise, rate limit errors), add suppression rules to filter them out automatically.

Running as a background service

macOS (launchd)

Use the launchd plist from the README. Key settings:

  • Set an explicit WorkingDirectory
  • Pass secrets via EnvironmentVariables
  • Configure StandardOutPath / StandardErrorPath for logs
  • Set KeepAlive = true for automatic restarts

Linux (systemd)

Use the systemd unit from the README. Key settings:

  • Set WorkingDirectory
  • Load secrets from an EnvironmentFile
  • Set Restart=on-failure for automatic restarts

Running with Docker

Compose (recommended)

docker compose up -d
docker compose logs -f claudear
docker compose down

Standalone

docker run -d --name claudear \
  -v $(pwd)/claudear.toml:/app/claudear.toml:ro \
  -v claudear-data:/app/data \
  -p 3100:3100 \
  ghcr.io/abnegate/claudear:latest start --poll --port 3100

Health check: GET /api/health returns 200 when Claudear is up.

Deployment checklist

  1. Run claudear seed so Claudear starts from now, not your entire backlog
  2. Confirm the dashboard loads at your configured port
  3. Manually trigger an issue with claudear trigger to confirm the full cycle works
  4. Tune max_concurrent and per-source limits based on your volume
  5. Enable regression monitoring and scheduled reports
